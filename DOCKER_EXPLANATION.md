# Docker в проекте Task Manager API

## Что такое Docker

Docker - это инструмент для запуска приложений в контейнерах. Контейнер можно представить как изолированную среду, внутри которой есть приложение, его зависимости, настройки запуска и минимальная операционная система.

Главная идея Docker: приложение запускается одинаково на разных компьютерах, потому что всё нужное описано в конфигурационных файлах проекта.

## Основные понятия

`Dockerfile` - инструкция для сборки образа. В нём описано, какой базовый образ взять, какие зависимости установить, какие файлы скопировать и какую команду запустить.

`Image` или образ - готовый шаблон контейнера. Например, из `Dockerfile` приложения собирается образ FastAPI-сервиса.

`Container` или контейнер - запущенный экземпляр образа. Один образ можно запустить много раз.

`docker-compose.yml` - файл, который описывает несколько контейнеров сразу и правила их совместной работы. В этом проекте Compose запускает приложение и MySQL.

`Network` - внутренняя сеть Docker. Контейнеры в одной сети могут обращаться друг к другу по имени сервиса. Здесь приложение обращается к базе по хосту `db`.

`Volume` - постоянное хранилище данных контейнера. Без volume данные MySQL исчезли бы после удаления контейнера. В этом проекте данные базы хранятся в volume `mysql_data`.

`.env` - файл с переменными окружения. В нём хранятся локальные значения вроде паролей и `SECRET_KEY`. Этот файл не добавляется в Git.

`.env.example` - безопасный пример `.env`, который можно сдавать и хранить в репозитории. В нём нет настоящих секретов.

## Как Docker устроен в этом проекте

Проект состоит из двух контейнеров:

- `app` - FastAPI-приложение `Task Manager API`
- `db` - база данных MySQL

Оба контейнера описаны в `docker-compose.yml`.

Контейнер `app` собирается из `Dockerfile` в корне проекта. Он:

- берёт базовый Python-образ
- устанавливает зависимости из `requirements.txt`
- копирует код приложения
- запускает `uvicorn main:app --host 0.0.0.0 --port 8000`

Контейнер `db` собирается из `docker/mysql/Dockerfile`. Он основан на официальном образе `mysql:8.4`.

## Почему секреты не лежат в Dockerfile

Секреты нельзя хранить в `Dockerfile`, потому что Dockerfile обычно попадает в репозиторий, а собранные образы могут передаваться другим людям или храниться в registry.

В этом проекте секреты передаются через `.env`:

- `MYSQL_PASSWORD`
- `MYSQL_ROOT_PASSWORD`
- `SECRET_KEY`

Файл `.env` добавлен в `.gitignore`, поэтому он не должен попадать в коммит. В репозитории хранится только `.env.example`.

## Как приложение подключается к базе

В `docker-compose.yml` для приложения задаётся переменная:

```text
DATABASE_URL=mysql+pymysql://${MYSQL_USER}:${MYSQL_PASSWORD}@db:3306/${MYSQL_DATABASE}
```

Здесь:

- `mysql+pymysql` - SQLAlchemy-драйвер для MySQL
- `${MYSQL_USER}` - пользователь базы из `.env`
- `${MYSQL_PASSWORD}` - пароль из `.env`
- `db` - имя сервиса базы данных внутри Docker-сети
- `3306` - порт MySQL внутри сети Docker
- `${MYSQL_DATABASE}` - имя базы данных

Когда FastAPI стартует, `database.py` читает `DATABASE_URL` из окружения, создаёт SQLAlchemy engine и подключается к MySQL.

## Как запустить проект в Docker

1. Создать локальный `.env` из примера:

```bash
cp .env.example .env
```

2. Для учебного запуска можно оставить демо-значения. Для своего пароля меняйте `.env` до первого запуска или пересоздавайте volume через `docker compose down -v`.

3. Собрать и запустить контейнеры:

```bash
docker compose up --build
```

4. Открыть веб-интерфейс:

```text
http://localhost:8000
```

Страницы интерфейса разделены:

- `http://localhost:8000/login` - вход
- `http://localhost:8000/register` - регистрация
- `http://localhost:8000/tasks-ui` - работа с задачами, фильтрами и сортировками

5. Swagger-документация API доступна здесь:

```text
http://localhost:8000/docs
```

## Как проверить CRUD

Через веб-интерфейс на `http://localhost:8000` можно:

- зарегистрировать пользователя
- войти
- создать задачу
- получить список задач
- сортировать и фильтровать список задач
- поменять статус задачи на `done`
- удалить задачу

Это покрывает CRUD:

- Create - `POST /tasks`
- Read - `GET /tasks`
- Update - `PUT /tasks/{task_id}`
- Delete - `DELETE /tasks/{task_id}`

## Как проверить сохранение данных через volume

1. Запустить проект:

```bash
docker compose up --build
```

2. Создать пользователя и задачу через `http://localhost:8000`.

3. Остановить и удалить контейнеры без удаления volume:

```bash
docker compose down
```

4. Снова поднять проект:

```bash
docker compose up
```

5. Войти тем же пользователем и проверить, что задача осталась в базе.

Данные сохраняются, потому что MySQL хранит их в volume `mysql_data`.

Если выполнить команду ниже, volume удалится и данные пропадут:

```bash
docker compose down -v
```

## Healthcheck и перезапуск контейнеров

В `docker-compose.yml` настроены healthcheck-и:

- `db` проверяется через `mysqladmin ping`
- `app` проверяется через `GET /healthz`

Также задано:

```yaml
restart: unless-stopped
```

Это означает, что Docker будет перезапускать контейнеры после сбоя, пока пользователь явно их не остановит.

В Compose также добавлен блок `deploy.restart_policy`, который полезен при запуске через Docker Swarm.

## Почему здесь два контейнера

Приложение и база данных разделены, потому что это ближе к реальной промышленной разработке:

- приложение можно обновлять отдельно от базы
- база хранит данные в отдельном volume
- контейнеры общаются по сети Docker
- конфигурация окружения отделена от кода

Такой подход проще переносить между локальной машиной, сервером и CI/CD.
